批量发布pod库

发表于 2018-03-03   |  

0x00 背景

每当一个开发周期完毕，app 发掉版本后，此次修改的 pod 库也需要发布，这时候人力一个一个去发，耗时耗力

于是弄个脚本来做这种事

0x01 前期准备

pod-template
标准化创建 pod 的一个模板，可以在官方的基础上增加自己的配置

fastlane

fastlane 是一个强大的工具，几乎承包了开发过程中需要用到的命令，pod 发布的命令他也集成了，我们可以通过他来实现 pod 的发布

由于每个 pod 都需要用到 fastlane 的功能，所以每个 pod 库中都需要一个 fastfile 文件，这个需求可以配置到上面说的 pod-template 中去，这样新建 pod 的时候就不用手动来创建这样一个文件了

需要用到的发布功能通过 fastlane 来提供 pod 发布的一些具体 action 功能，如 pod lint、version increase、pod push，我们写在一个统一的文件里

当你写好这样一个 fastfile 文件的时候，我们不需要 copy 到每个 pod 库中，我们使用 fastlane 的 import 功能来使每个子 pod 库复用这份文件，这份文件会放在远程某个 git 中，将来需要增加修改功能，也只要在这份文件中修改，引用了这份文件的子 pod 就能使用到正确的功能。

0x02 最终方案

0x0201 shell 脚本

脚本思路是这样

  1. 从主工程 podfile 中筛选出需要发布的 pod，存放到一个数组中
  2. 在主工程文件下生成一个 podRepos 文件夹用来存放 pod repo，循环3-8步
  3. 获取数组中的 pod 信息，将 pod clone 下来，进入对应的 pod 目录
  4. 检出响应的 feature 分支，执行 pod lint ，通过则继续，不通过则回到第3步继续下一个 pod
  5. 合到 develop，合到 master
  6. increse version
  7. pod push
  8. develop 合 master
  9. 回到 podRepos 目录
  10. 所有发完后删除 podRepos 文件

这里脚本还差最后一步没有实现，
所有 pod 发完后，主工程 podfile 文件应该将对应 pod 的 feature 指定给删掉

0x0202 每个子pod库路径下的 fastfile 文件

由于通过 import 的方式来引用提供主要功能的 fastfile 文件，子 pod 下的这个 fastfile 文件就相当的简洁了

0x0203 发布 pod fastfile 文件

这个文件正真包含了我们封装的发布pod所需 action
