发表于 2016-05-16   |  

运输层

传递应用层报文

主要协议

  • TCP ：面向连接；长报文划分成短报文；流量控制；确保传递拥塞控制机制（网络拥塞、抑制传输速率）
  • UDP ：无连接服务；不可靠；无拥塞机制；无流量控制

分组 ： 报文段

MSS : 在TCP连接建立时，收发双方协商通信时每一个报文段所能承载的最大数据长度
RTT : 发送端发送数据开始，到发送端收到来自接收端的确认（接收端收到数据后便立即发送确认），总共经历的时延
RTO : 从发送端发送数据开始经历可以判定包以丢失的时长

流量控制 VS 拥塞机制

在没看书之前呢，我一直把这两个概念混淆起来，觉得他们是一个意思，但是，其实不是嗒啊。

流量控制（TCP滑动窗口）

图片来源

上图便是著名的TCP滑窗的示意图

  • Category #1 代表已收到接收方ACK的分组；
  • Category #2 代表已发送但还没收到ACK的分组；
  • Category #3 代表暂时未发送的分组（接受方的缓冲区还有空间）；
  • Category #4 暂时不能发送的数据（接收方的缓冲区放不下啦）

在数据发送时，发送端和接收端会有一个滑窗，这个滑窗显示了接收方好友多少空间能用于接收数据，以此实现发送方能根据接收方的接收能力来控制每次发送的数据量

当然事实实现起来没有我说的那么简单，里面包含了各种情况的考虑，比如说Category #3为0时是否关闭TCP连接的判断、接受方缓冲区剩余空间很小时的优化等等

拥塞控制（TCP拥塞窗口）

这里只简单介绍一下拥塞控制主要的四个算法，拥塞窗口简写为cwnd

  1. 慢启动
    cwnd大小初始为1个MSS大小，每收到一个ACK cwnd++，每过一个RTT cwnd*2，达到上限sshthresh后进入拥塞避免
  2. 拥塞避免
    每收到一个 cwnd = cwnd + 1/cwnd；每过一个RTT cwnd++
  3. 拥塞发生
    • 超时重传数据包 sshthresh = cwnd /2； cwnd = 1；进入慢启动
    • 收到三个相同ACK cwnd = cwnd /2；sshthresh = cwnd；进入快速恢复
  4. 快速恢复 cwnd = sshthresh + 3 * MSS
    • 如果在收到另一个三个相同的ACK cwnd = cwnd +1
    • 如果收到新的ACK cwnd = sshthresh，进入拥塞避免

个人对两者的理解如下

  • 流量控制是通过滑窗中剩余可发送数据大小来控制发送方数据的发送，避免一个TCP连接一下子发送大量的数据占用带宽。
  • 拥塞机制是结合不同的算法对拥塞窗口的大小做改变，来适应一些突发的网络状况不佳的情况，比如：网络突然拥挤，造成大量的丢包需要重传，这时候重新设置滑窗的大小便可以减小重发的数据的大小，避免进一步的阻塞带宽

发送方取拥塞窗口与滑动窗口中的最小值作为发送上限

重传机制

书上介绍两种

  • 退回N步 : 发送方重发最后一个正确顺序接收的信息帧之后的所有未被确认的帧，比如发送方按1、2、3、4、5发送，而接收方收到了1、3、5，这时候发送方会重发2、3、4、5，接收方会丢弃之前收到的3、5重新接受
  • 选择重传 : 只重传丢失或超时的包，避免重复传送那些本来已经正确到达接收端的数据，比如发送方按1、2、3、4、5发送，而接收方收到了1、3、5，这时候发送方会重发2、4，接收方之前收到的3、5会被保留

Socket

socket是应用层和运输层通讯的中间接口，应用层通过socket将报文发送到运输层，运输层接到报文段拆包成报文，通过socket传输到应用层。socket分辨着引用层的不同进程所使用的端口，以此进行数据传输。

基于TCP的socket客户端-服务器端连接需要双方约定使用相同的端口，因为tcp的socket使用四元组（源端口，源IP，目的端口，目的IP），服务器收到数据后会为不同的源端口源地址分配不同的socket（即使目的端口和目的IP一致）所以客户端要讲数据正确投到服务器某个进程的话，要使用约定好的端口。

基于UDP的socket不用如此，因为UDP的socket使用二元组（目的端口，目的IP）。
