UIResponder、UIView、UIControl

三者的继承关系 UIControl -> UIView -> UIResponder

UIResponder

UIResponder称为响应者对象，能够接受并处理UIEvent

提供诸如 touchesBegan、touchesMoved、touchesEnded 等方法

UIView

UIView作为UIResponder的子类，可以通过重写touchesBegan、touchesMoved、touchesEnded等方法来处理时间

UIControl

UIControl是UIView的子类，我们平时使用的UIButton、UISwitch的直接父类便是UIControl。

UIControl对touchesBegan、touchesMoved、touchesEnded做一次封装，提供诸如UIControlEventTouchUpInside、UIControlEventTouchUpOutside的事件，让对控件的触摸事件使用更加方便

响应链

屏幕上的View对用户触摸事件的响应大致分为两个步骤：

  • 定位最终响应事件的View
  • 按照响应链找最终能响应事件的控件

当一个用户触摸屏幕，响应过程便开始了

在查找View的时候调用了两个重要方法

寻找最适合的响应View

寻找顺序

AppDelegate -> UIApplication -> UIWindow -> UIViewController -> UIView

控件调用hitTest:withEvent:，该方法又调用pointInside:withEvent:方法。当pointInside方法判断触摸点是否在view内并返回结果，hitTest方法根据返回的Bool值做不同的操作，如果返回Yes，则对自己的子View逐层调用hitTest方法得到最终能响应的View（如子View都返回No则返回Self）；如果返回No，则直接返回nil。以上找到了最终最适合的View。

根据响应链响应事件

当找到最适合的View之后，便开始让控件响应事件

响应顺序正好与寻找顺序相反

UIView -> UIViewController -> UIWindow -> UIApplication -> AppDelegate

从最适合View开始逐层往上找父View再到Controller。。。。。找到事件正真的响应者处理事件。

Key-Value Coding键值编码机制。

已KVC给属性设值为例setValue:forKey:，编译如下

一个对象在调用setValue的时候执行的步骤

  • 根据方法名找到所需要的SEL
  • 在自己isa指针所指向的类对象里结合SEL，找到具体的方法实现的IMP
    • 先找setKey:方法
    • setter方法没找到，如果类方法accessInstanceVariablesDirectly返回YES。那么按 _key，_isKey，key，iskey的顺序搜索成员名
    • 如果没有找到成员变量，调用setValue:forUnderfinedKey:
  • 调用IMP

KVO

Key-Value Observing 让一个对象能够观察另一个对象的属性，当属性改变便收到通知，然后观察者做一些操作。具体的使用就不说了，是观察者模式的一种实施方案（还有通知、delegate）。

内部实现

被观察对象的类会被重写，runtime生成一个伪装类来代替原来的类，如果你查看对象的isa指针，会发现isa和原来相比会多一个NSKVONotifying_的前缀。

这个伪装类会重写一些方法，最重要的便是被观察的属性的setter方法，在setter方法调用了willChangeValueForKey和didChangeValueForKey来做通知。

当然为了防止重复生成伪装类，原来的类会增加一个_isKVOA来标识这个类是否已被动态生成过KVO伪装类。

手动通知

通常情况下，从NSObject继承而来的类都具有KVO自动通知，不需要额外写代码。如果需要手动控制通知（可以减少不必要的通知），那么需要重写automaticallyNotifiesObserversForKey:，为手动通知的属性返回NO。同时用willChangeValueForKey和didChangeValueForKey来触发通知

Tip

不通过访问器来改变属性将不能触发通知

RunLoop

RunLoop一直以来感觉是个神秘的东西，个人没有实战用过。。。但是不影响去了解一下他的必要性。。。。

个人觉得他的作用就是起一个Loop循环，让App一直跑着。

RunLoop VS Thread

首先线程和RunLoop一对一存在，但线程不一定有RunLoop，而主线程会默认起一个RunLoop，这个很好理解，主线程如果没有RunLoop，代码一跑完App就自己关了呗。

线程 和 RunLoop 之间的一对一关系是由一个全局的 Dictionary 维护。线程创建时并不会获取 RunLoop，如果你不主动获取，那RunLoop一直都不会被创建。当你第一次获取时，RunLoop才被创建，线程结束时会销毁RunLoop。

当你需要一个线程常驻，就可以给线程获取一个RunLoop，通过CFRunLoopGetMain() 和 CFRunLoopGetCurrent()

RunLoop 内部结构

CFRunLoopRef

CFRunLoopModeRef : 每个Mode都有一个ModeName，你只能通过ModeName操作Mode,默认预置的Mode有 kCFRunLoopDefaultMode (App平常运行的状态) 、 UITrackingRunLoopMode(追踪 ScrollView 的滑动) 、 kCFRunLoopCommonModes (操作 commonModeItems，或标记一个 Mode 为 Common)

CFRunLoopSourceRef : 有Source0 和 Source1两种。

  • Source0 只包含了一个回调，不能主动触发事件。使用时，先调用 CFRunLoopSourceSignal(source)，将 Source 标记为待处理，然后手动调用 CFRunLoopWakeUp(runloop) 来唤醒 RunLoop处理事件

  • Source1 包含了一个 mach_port 和一个回调（函数指针），被用于通过内核和其他线程相互发送消息，能主动唤醒 RunLoop 的线程

CFRunLoopTimerRef : 基于时间的触发器

CFRunLoopObserverRef : 观察者，包含回调，当 RunLoop 的状态发生变化时，观察者能通过回调接受变化

他们之间的关系是：一个 RunLoop 有若干个 Mode，每个 Mode 又由若干个 Source/Timer/Observer 这样的 item组成

CommonModes

个人认为这个是平时最常会用到的东西 : 一个 Mode 可以将自己标记为Common属性（通过将其 ModeName 添加到 RunLoop 的 commonModes 中）。每当 RunLoop 的内容发生变化时，RunLoop 都会自动将 commonModeItems 里的 Source/Observer/Timer 同步到具有 Common 标记的所有Mode里。

kCFRunLoopDefaultMode 和 UITrackingRunLoopMode都已经被标记为Common属性

实际用到便是把 NSTimer 加到 commonModeItems ，这样在滑动时 Timer 依然能有用

RunLoop相关实例

AutoreleasePool

即将进入RunLoop时会调用 _objc_autoreleasePoolPush() 创建自动释放池。

准备进入休眠 时调用_objc_autoreleasePoolPop() 和 _objc_autoreleasePoolPush() 释放旧的池并创建新池

即将退出RunLoop时调用 _objc_autoreleasePoolPop() 来释放自动释放池。

界面Update

当在操作UI时，或者手动调用setNeedsLayout、setNeedsDisplay方法后，这个 UIView、CALayer 就被标记为待处理，并被提交到一个全局的容器去。

即将进入休眠 和 即将退出Loop 时，回调去执行一个函数遍历所有待处理的 UIView、CALayer 以执行实际的绘制和调整，更新 UI 界面。

继承的访问权限

首先三种访问关键字public、protected、private，访问权限如下

public ：在哪里都可以访问

protected ：只能在本类和子类访问

private ：只能在本类访问

Class继承默认是private

父类中成员变量访问权限 继承方式 子类中成员变量访问权限
public public public
public protected protected
public private private
protected public protected
protected protected protected
protected private private
private public 子类不能访问
private protected 子类不能访问
private private 子类不能访问
